# Теория для Дня 1: Введение в QA и базовые понятия

## 1. Что такое QA и чем отличается от разработки

### Что такое QA (Quality Assurance)?

**QA (Quality Assurance)** — это процесс обеспечения качества программного обеспечения. QA-специалист (тестировщик) отвечает за проверку того, что программное обеспечение работает корректно, соответствует требованиям и ожиданиям пользователей.

**Основные задачи QA:**
- Находить ошибки (баги) в программном обеспечении
- Проверять соответствие функциональности требованиям
- Обеспечивать качество пользовательского опыта
- Предотвращать появление дефектов на ранних этапах разработки
- Документировать найденные проблемы

### Отличие QA от разработки

| **Разработка (Development)** | **QA (Quality Assurance)** |
|-------------------------------|----------------------------|
| Создает функциональность | Проверяет функциональность |
| Пишет код | Тестирует код |
| Реализует требования | Проверяет соответствие требованиям |
| Исправляет баги после их обнаружения | Находит баги до релиза |
| Фокус на "как сделать" | Фокус на "что может сломаться" |

**Важно:** QA и разработка работают вместе в команде. Разработчик создает продукт, а тестировщик помогает убедиться, что продукт качественный и готов к использованию.

## 2. Разница между Bug (дефект) и Issue

### Bug (Дефект)

**Bug (баг, дефект)** — это ошибка в программном обеспечении, которая приводит к неправильному поведению программы.

**Характеристики бага:**
- Программа работает не так, как ожидалось
- Есть четкое ожидаемое поведение и фактическое поведение
- Можно воспроизвести (повторить)
- Требует исправления разработчиком

**Примеры багов:**
- Кнопка "Отправить" не работает при нажатии
- Форма принимает некорректный email адрес
- Страница не загружается в определенном браузере
- Текст отображается неправильно (обрезается или накладывается)

### Issue (Проблема/Задача)

**Issue** — более широкое понятие, которое включает в себя не только баги, но и другие типы задач.

**Типы Issues:**
1. **Bug** — ошибка в коде
2. **Feature Request** — запрос на новую функциональность
3. **Improvement** — улучшение существующей функциональности
4. **Task** — техническая задача (например, обновление документации)
5. **Question** — вопрос или уточнение

**Разница:**
- **Bug** — это всегда проблема, ошибка, которую нужно исправить
- **Issue** — это любая задача в системе управления проектом (может быть багом, но может быть и запросом на новую функцию)

**В контексте работы:**
- Когда находите ошибку → создаете **Bug**
- Когда нужно что-то улучшить или добавить → создаете **Issue** (тип: Feature Request или Improvement)

## 3. Тест-планы, тест-кейсы, чек-листы

### Тест-план (Test Plan)

**Тест-план** — это документ, описывающий стратегию тестирования проекта.

**Что включает тест-план:**
- Цели и задачи тестирования
- Области тестирования (что будет тестироваться)
- Области, которые не будут тестироваться
- Подход к тестированию
- Критерии начала и окончания тестирования
- Ресурсы (люди, инструменты, время)
- Риски и способы их минимизации

**Пример структуры тест-плана:**
```
1. Введение
2. Область тестирования
3. Подход к тестированию
4. Критерии входа/выхода
5. Ресурсы
6. Расписание
7. Риски
```

### Тест-кейс (Test Case)

**Тест-кейс** — это документированная последовательность шагов для проверки конкретной функциональности.

**Структура тест-кейса:**
1. **ID** — уникальный идентификатор
2. **Название** — краткое описание того, что проверяется
3. **Предусловия** — что должно быть выполнено перед тестом
4. **Шаги** — пошаговая инструкция
5. **Ожидаемый результат** — что должно произойти
6. **Фактический результат** — что произошло на самом деле (заполняется после выполнения)
7. **Статус** — Pass/Fail/Blocked

**Пример тест-кейса:**

| Поле | Значение |
|------|----------|
| **ID** | TC-001 |
| **Название** | Проверка отправки формы с валидными данными |
| **Предусловия** | Пользователь находится на странице формы обратной связи |
| **Шаги** | 1. Ввести имя "Иван"<br>2. Ввести email "ivan@example.com"<br>3. Ввести сообщение "Тестовое сообщение"<br>4. Нажать кнопку "Отправить" |
| **Ожидаемый результат** | Форма успешно отправлена, отображается сообщение "Спасибо за обращение!" |
| **Фактический результат** | (заполняется после выполнения) |
| **Статус** | Pass/Fail |

### Чек-лист (Checklist)

**Чек-лист** — это список пунктов для проверки без детального описания шагов.

**Отличие от тест-кейса:**
- **Чек-лист** — краткий список того, что нужно проверить
- **Тест-кейс** — детальная инструкция с шагами

**Когда использовать:**
- **Чек-лист** — для быстрой проверки, exploratory тестирования, smoke-тестов
- **Тест-кейс** — для детального тестирования, регрессионных тестов, когда нужна точная документация

**Пример чек-листа для формы регистрации:**
```
☐ Поле "Имя" принимает текст
☐ Поле "Email" валидирует формат email
☐ Поле "Пароль" скрывает введенный текст
☐ Кнопка "Зарегистрироваться" активна при заполнении всех полей
☐ Сообщение об ошибке отображается при пустых полях
☐ Форма отправляется с валидными данными
☐ Форма не отправляется с невалидными данными
```

## 4. Роли QA: Manual vs Automation, уровни (Junior/Middle/Senior)

### Manual QA vs Automation QA

#### Manual QA (Ручное тестирование)

**Manual QA** — тестировщик, который выполняет тесты вручную, без написания кода.

**Что делает Manual QA:**
- Выполняет тест-кейсы вручную
- Проводит exploratory тестирование
- Проверяет UI/UX
- Тестирует функциональность, нажимая кнопки, заполняя формы
- Находит баги через ручное взаимодействие с приложением

**Плюсы:**
- Можно начать без знания программирования
- Гибкость в тестировании
- Понимание пользовательского опыта
- Быстрое начало работы

**Минусы:**
- Медленнее для повторяющихся тестов
- Требует больше времени для регрессионного тестирования

#### Automation QA (Автоматизированное тестирование)

**Automation QA** — тестировщик, который пишет код для автоматического выполнения тестов.

**Что делает Automation QA:**
- Пишет скрипты для автоматизации тестов
- Использует инструменты (Selenium, Cypress, Playwright и др.)
- Автоматизирует повторяющиеся тесты
- Интегрирует тесты в CI/CD процесс

**Плюсы:**
- Быстрое выполнение повторяющихся тестов
- Возможность запускать тесты в любое время
- Экономия времени на регрессионном тестировании

**Минусы:**
- Требует знания программирования
- Дольше настраивать автоматизацию
- Не все тесты можно автоматизировать

**Важно:** Большинство QA начинают с Manual тестирования, а затем изучают автоматизацию.

### Уровни QA специалистов

#### Junior QA (Начинающий)

**Опыт:** 0-1 год

**Навыки:**
- Понимание базовых концепций тестирования
- Умение находить баги
- Написание простых тест-кейсов и баг-репортов
- Работа с инструментами для баг-трекинга (Trello, Jira)
- Базовое использование DevTools

**Задачи:**
- Выполнение тест-кейсов
- Поиск багов
- Документирование найденных проблем
- Работа под руководством более опытных коллег

#### Middle QA (Средний уровень)

**Опыт:** 1-3 года

**Навыки:**
- Глубокое понимание процессов тестирования
- Умение составлять тест-планы и стратегии
- Знание различных видов тестирования
- Работа с API тестированием (Postman)
- Базовые навыки автоматизации (опционально)
- Работа с базами данных

**Задачи:**
- Планирование тестирования
- Составление тест-планов
- Координация тестирования в команде
- Работа с требованиями
- Менторинг Junior QA

#### Senior QA (Старший уровень)

**Опыт:** 3+ лет

**Навыки:**
- Экспертиза во всех областях тестирования
- Умение строить стратегию тестирования для проекта
- Навыки автоматизации (написание фреймворков)
- Работа с CI/CD
- Технические навыки (SQL, API, базовые знания языков программирования)
- Лидерские качества

**Задачи:**
- Построение процессов тестирования в команде
- Техническое руководство
- Принятие решений о стратегии тестирования
- Менторинг команды
- Взаимодействие с заказчиками и стейкхолдерами

## 5. Основные принципы тестирования (ISTQB)

### Принципы тестирования по ISTQB

**ISTQB (International Software Testing Qualifications Board)** — международная организация, которая устанавливает стандарты в области тестирования.

**7 основных принципов тестирования:**

1. **Тестирование показывает наличие дефектов**
   - Тестирование может показать, что в программе есть дефекты, но не может доказать их отсутствие
   - Даже если тесты прошли успешно, это не гарантирует отсутствие багов

2. **Исчерпывающее тестирование невозможно**
   - Невозможно протестировать все возможные комбинации входных данных
   - Нужно использовать техники тест-дизайна для выбора наиболее важных тестов

3. **Раннее тестирование**
   - Тестирование должно начинаться как можно раньше в жизненном цикле разработки
   - Нахождение багов на ранних этапах дешевле, чем на поздних

4. **Скопление дефектов**
   - Большинство дефектов находится в небольшом количестве модулей
   - Если в модуле найдено много багов, вероятно, там их еще больше

5. **Остерегайтесь пестицидов**
   - Если одни и те же тесты запускаются многократно, они перестают находить новые дефекты
   - Тесты нужно регулярно пересматривать и обновлять

6. **Тестирование зависит от контекста**
   - Разные типы приложений требуют разных подходов к тестированию
   - Веб-приложение тестируется иначе, чем мобильное приложение

7. **Заблуждение об отсутствии ошибок**
   - Отсутствие ошибок не означает, что система готова к использованию
   - Система может быть бесполезной, если не соответствует требованиям пользователей

## 6. Цели и задачи QA

### Основные цели QA

1. **Обеспечение качества продукта**
   - Продукт должен работать корректно
   - Продукт должен соответствовать требованиям

2. **Предотвращение дефектов**
   - Нахождение проблем на ранних этапах
   - Снижение количества багов в продакшене

3. **Защита репутации компании**
   - Качественный продукт повышает доверие пользователей
   - Избежание негативных отзывов из-за багов

4. **Экономия ресурсов**
   - Исправление багов на ранних этапах дешевле
   - Предотвращение проблем до релиза

### Задачи QA специалиста

**Ежедневные задачи:**
- Выполнение тест-кейсов
- Поиск и документирование багов
- Проверка исправленных багов (ретест)
- Участие в планировании и обсуждениях

**Периодические задачи:**
- Составление тест-планов
- Написание тест-кейсов
- Анализ требований
- Подготовка отчетов о тестировании

**Долгосрочные задачи:**
- Улучшение процессов тестирования
- Изучение новых инструментов и технологий
- Развитие навыков автоматизации

## 7. Типы тестирования (краткий обзор)

### По уровню тестирования

1. **Unit тестирование** — тестирование отдельных компонентов кода (выполняется разработчиками)
2. **Integration тестирование** — тестирование взаимодействия между компонентами
3. **System тестирование** — тестирование всей системы в целом
4. **Acceptance тестирование** — тестирование на соответствие требованиям заказчика

### По типу выполнения

1. **Manual тестирование** — выполняется вручную
2. **Automated тестирование** — выполняется автоматически с помощью скриптов

### По целям тестирования

1. **Функциональное тестирование** — проверка функциональности
2. **Нефункциональное тестирование** — проверка производительности, безопасности, удобства использования

### По времени выполнения

1. **Smoke тестирование** — быстрая проверка основных функций
2. **Regression тестирование** — проверка того, что новые изменения не сломали существующую функциональность
3. **Sanity тестирование** — быстрая проверка после исправления багов

### По способу тестирования

1. **Exploratory тестирование** — исследовательское тестирование без заранее подготовленных тест-кейсов
2. **Ad-hoc тестирование** — случайное тестирование без плана

## 8. Инструменты для начинающего QA

### Управление багами и задачами

1. **Trello** — простой инструмент для управления задачами
   - Создание досок и карточек
   - Отслеживание статусов задач
   - Бесплатный план доступен

2. **Jira** — профессиональный инструмент для управления проектами
   - Создание багов и задач
   - Отслеживание спринтов
   - Интеграция с другими инструментами

3. **TestRail** — специализированный инструмент для управления тест-кейсами
   - Хранение тест-кейсов
   - Отслеживание выполнения тестов
   - Генерация отчетов

### Браузерные инструменты

1. **Chrome DevTools** — встроенные инструменты разработчика
   - Проверка HTML элементов
   - Просмотр ошибок в консоли
   - Анализ сетевых запросов
   - Эмуляция мобильных устройств

2. **Firefox Developer Tools** — аналогичные инструменты для Firefox

### Инструменты для скриншотов

1. **Lightshot** — быстрый захват скриншотов с возможностью аннотаций
2. **Nimbus Screenshot** — расширение для браузера с редактированием
3. **Snipping Tool** — встроенный инструмент Windows

### API тестирование (для будущего изучения)

1. **Postman** — инструмент для тестирования API
2. **Insomnia** — альтернатива Postman

### Автоматизация (для будущего изучения)

1. **Selenium** — фреймворк для автоматизации веб-приложений
2. **Cypress** — современный инструмент для E2E тестирования
3. **Playwright** — инструмент от Microsoft для автоматизации
