# Теория для Дня 2: Баг-репорты и жизненный цикл дефекта

## 1. Что такое баг-репорт и зачем он нужен

### Что такое баг-репорт?

**Баг-репорт** — это документ, который описывает найденную ошибку так, чтобы разработчик мог её понять и исправить.

### Зачем нужен баг-репорт?

Баг-репорт выполняет три основные функции:

1. **Разработчик должен понять, что сломалось**
   - Четкое описание проблемы
   - Где именно произошла ошибка
   - Что именно не работает

2. **Разработчик должен воспроизвести баг**
   - Пошаговые инструкции
   - Конкретные данные для ввода
   - Условия, при которых баг появляется

3. **Разработчик должен понять, как это должно работать**
   - Ожидаемое поведение
   - Фактическое поведение
   - Разница между ними

### Пример: Плохой баг-репорт vs Хороший баг-репорт

**❌ Плохой баг-репорт:**
```
Заголовок: Ошибка
Описание: Не работает
```

**✅ Хороший баг-репорт:**
```
Заголовок: Кнопка "Отправить" не работает на странице формы обратной связи

Описание:
При нажатии на кнопку "Отправить" форма не отправляется, 
не показывается сообщение об успехе.

Шаги воспроизведения:
1. Открыть страницу формы обратной связи
2. Заполнить все обязательные поля
3. Нажать кнопку "Отправить"

Ожидаемый результат: Форма отправляется, показывается сообщение "Успешно!"

Фактический результат: Форма не отправляется, ничего не происходит
```

## 2. Структура баг-репорта

### Основные элементы баг-репорта:

#### 1. Заголовок (Title)

**Заголовок** — краткое описание проблемы (что сломалось, где).

**Правила написания заголовка:**
- Конкретный и информативный
- Указывает, что сломалось
- Указывает, где произошла ошибка

**Примеры:**

❌ **Плохо:**
- "Ошибка"
- "Не работает"
- "Проблема с формой"

✅ **Хорошо:**
- "Кнопка 'Отправить' не работает на странице формы обратной связи"
- "Email принимает некорректный формат в форме регистрации"
- "Сообщение об ошибке не отображается при пустом поле 'Имя'"

#### 2. Описание (Description)

**Описание** — что произошло, простым языком.

**Что должно быть в описании:**
- Что именно не работает
- Когда это происходит
- Какие данные использовались
- Приоритет (Высокий/Средний/Низкий) и почему

**Пример:**
```
Форма обратной связи принимает email адрес без доменной части 
(например, "test@") как валидный. Это может привести к проблемам 
при отправке писем.

Приоритет: Высокий (критичная функция не работает правильно)
```

#### 3. Шаги воспроизведения (Steps to Reproduce)

**Шаги воспроизведения** — пошаговая инструкция, чтобы другой человек мог повторить баг.

**Правила написания шагов:**
- Каждый шаг — одно действие
- Конкретные данные (не "ввести данные", а "ввести email: test@")
- Нумерация шагов
- Понятные формулировки

**Пример:**
```
Шаги воспроизведения:
1. Открыть страницу с формой обратной связи
2. Найти поле "Email"
3. Ввести некорректный email: "test@"
4. Убрать фокус с поля (кликнуть вне поля)
5. Попытаться отправить форму
```

#### 4. Ожидаемый результат (Expected Result)

**Ожидаемый результат** — что должно произойти при правильной работе.

**Пример:**
```
Ожидаемый результат:
- Система должна отобразить ошибку: "Введите корректный email адрес"
- Форма не должна отправляться
```

#### 5. Фактический результат (Actual Result)

**Фактический результат** — что произошло на самом деле.

**Пример:**
```
Фактический результат:
- Форма принимает "test@" как валидный email
- Валидация проходит успешно
- Форма может быть отправлена с некорректными данными
```

#### 6. Приоритет (Priority)

**Приоритет** — важность бага для пользователя и системы.

**Критерии определения приоритета:**

**Высокий (High):**
- Блокирует работу пользователя
- Критичная функция не работает
- Невозможно использовать основную функциональность
- Пример: Кнопка "Войти" не работает — пользователь не может войти в систему

**Средний (Medium):**
- Функция работает неправильно, но есть обходной путь
- Не блокирует основную работу
- Пример: Форма принимает некорректный email, но можно ввести корректный

**Низкий (Low):**
- Косметическая проблема
- Опечатка в тексте
- Не влияет на функциональность
- Пример: Неправильная пунктуация в сообщении об ошибке

#### 7. Скриншоты (Screenshots)

**Скриншоты** — визуальное подтверждение бага.

**Зачем нужны скриншоты:**
- Показывают проблему визуально
- Помогают разработчику быстрее понять проблему
- Документируют состояние на момент обнаружения бага

**Как делать скриншоты:**
- Использовать инструменты: Lightshot, Nimbus Screenshot, Snipping Tool
- Добавлять аннотации (стрелки, текст) для указания проблемного места
- Сохранять в понятном формате (PNG, JPG)

## 3. Жизненный цикл дефекта

### Что такое жизненный цикл дефекта?

**Жизненный цикл дефекта** — это последовательность статусов, через которые проходит баг от момента обнаружения до закрытия.

### Статусы жизненного цикла:

#### 1. New (Новый)

**New** — баг найден, создан баг-репорт.

**Что происходит:**
- QA находит баг
- Создает баг-репорт в системе (Trello, Jira)
- Баг находится в статусе "New"

**Пример:**
```
Статус: New
Комментарий: "Найден баг в валидации email"
```

#### 2. Open (Открыт)

**Open** — разработчик взял баг в работу.

**Что происходит:**
- Разработчик видит баг в системе
- Берет баг в работу
- Баг переходит в статус "Open"

**Пример:**
```
Статус: Open
Комментарий: "Взят в работу разработчиком"
```

#### 3. Fixed (Исправлен)

**Fixed** — разработчик исправил баг.

**Что происходит:**
- Разработчик исправил код
- Баг переходит в статус "Fixed"
- Готов к проверке QA (ретест)

**Пример:**
```
Статус: Fixed
Комментарий: "Исправлено в версии 1.1"
```

#### 4. Closed (Закрыт)

**Closed** — QA проверил, что баг исправлен (ретест прошел).

**Что происходит:**
- QA проверяет исправление (ретест)
- Убеждается, что баг больше не воспроизводится
- Баг переходит в статус "Closed"

**Пример:**
```
Статус: Closed
Комментарий: "Ретест пройден, баг закрыт"
```

#### 5. Reopened (Переоткрыт)

**Reopened** — баг не исправлен, нужно вернуть разработчику.

**Что происходит:**
- QA проверяет исправление (ретест)
- Баг все еще воспроизводится
- Баг переходит в статус "Reopened"
- Возвращается разработчику

**Пример:**
```
Статус: Reopened
Комментарий: "Баг все еще воспроизводится, требуется доработка"
```

### Схема жизненного цикла:

```
New → Open → Fixed → Closed
                ↓
            Reopened → Open → Fixed → Closed
```

### Важные моменты:

1. **Ретест обязателен** — QA должен проверить каждое исправление
2. **Комментарии важны** — на каждом этапе нужно добавлять комментарии
3. **Статусы помогают отслеживать** — видно, на каком этапе находится каждый баг

**Что вы узнали:**
- Баг-репорт — это документ, описывающий ошибку для разработчика
- Структура баг-репорта: заголовок, описание, шаги, ожидаемый/фактический результат, приоритет, скриншоты
- Приоритет бага: Высокий (блокирует работу), Средний (есть обходной путь), Низкий (косметическая проблема)
